# 架构设计文档

## 1. 整体架构

PEC 网站采用**前后端分离 + 内网穿透**架构，前端纯静态托管在 CDN，后端跑在协会群晖 NAS 内网，通过 Cloudflare Tunnel 暴露 API。

│                  🌍 外网用户层                                  │
│  招生访客 / 协会成员 / 同学家长 (浏览器)                          │
└────────────────────────┬─────────────────────────────────────┘
                         │ HTTPS
                         ▼
┌──────────────────────────────────────────────────────────────┐
│           📦 前端静态托管层（GitHub Pages / CF Pages）           │

│  ├─ 公开页面：首页 / 团队 / 项目 / 招新 / 活动                    │
│  └─ 半公开页面：设备状态 / 物品清单（含留言）                       │
└────────────────────────┬─────────────────────────────────────┘
                         │ AJAX / SSE
                         ▼
┌──────────────────────────────────────────────────────────────┐

## 2. 数据流详解

### 2.1 设备状态采集（拓竹打印机）

```
拓竹 A1/X2D
   ↓ LAN Mode MQTT (mqtts://printer-ip:8883)
Bambu MQTT Collector 容器
   ↓ 解析状态：IDLE / PRINTING / OFFLINE
   ↓ HTTP POST /api/devices/{id}/state
Hono API 服务
   ↓ 写入 device_states 表
   ↓ 通过 SSE 通道推送
前端订阅 /api/events 的客户端
   ↓ 实时更新 UI 卡片
```

### 2.2 设备状态采集（树莓派）

```
树莓派启动 pi-agent.sh (systemd timer)
   ↓ 每 30 秒
   ↓ HTTP POST /api/devices/{id}/heartbeat
   ↓ Body: {cpu, mem, temp, uptime}
Hono API
   ↓ 更新 last_seen 时间戳
   ↓ 超过 90s 未上报 → 标记离线
```

### 2.3 物品留言流程

```
用户在物品详情页点"留言"
   ↓ 弹出表单：部门口令 + 学号 + 姓名 + 内容
   ↓ POST /api/messages
Hono API
   ↓ 校验部门口令（bcrypt 对比）
   ↓ 失败 → 401
   ↓ 成功 → 写入 messages 表
前端刷新留言列表（不刷整页）
```

## 3. 安全模型

### 3.1 信任分级

| 角色 | 能做什么 | 不能做什么 |
|------|---------|----------|
| **匿名访客** | 浏览所有公开页 + 看设备状态/物品状态 | 留言、删除、修改任何数据 |
| **持口令成员** | 留言（带学号+姓名） | 删除他人留言、改设备配置 |
| **主人（管理员）** | 全部操作 + 后台管理 | - |

### 3.2 部门口令机制

- 后台为每个部门生成随机 8 字符口令（如 `a8x7k9m2`）
- 数据库存 bcrypt 哈希，明文只在生成时显示给主人
- 主人通过私下渠道告知部门成员
- 任何部门口令均可留言，留言上**显示用户自报的学号+姓名**
- 学号格式校验：`^\d{4}[A-Z0-9]{4,10}$`（不验真）
- 主人后台可一键重置某部门口令（处理泄漏）

### 3.3 防滥用

- 留言接口速率限制：同 IP 每分钟 ≤ 3 次（Hono rate-limiter）
- 留言内容长度上限：500 字
- XSS 防护：前端用 DOMPurify 清洗
- SQL 注入防护：Drizzle ORM 参数化查询
- CORS：仅允许前端域名

### 3.4 主人后台认证

- 单一管理员账号（环境变量 `ADMIN_PASSWORD_HASH`）
- 登录后下发 JWT（7 天有效期）
- 所有 `/admin/*` 路由 middleware 校验

## 4. 关键设计决策

### 4.1 为什么前端用静态 SSG？

- ✅ 公开页面 SEO 友好（招新时同学能搜到）
- ✅ 托管成本为 0（GitHub Pages 免费）
- ✅ CDN 加速天然支持
- ⚠️ 动态部分（设备状态、留言）通过 AJAX 调后端补足

### 4.2 为什么用 SSE 而不是 WebSocket？

- 设备状态是**单向推送**（后端 → 前端），不需要双向通信
- SSE 基于 HTTP，更易穿透 Cloudflare Tunnel
- 浏览器原生 `EventSource` API，无需额外库
- 断线自动重连，开发成本低

### 4.3 为什么用 SQLite 而不是 PostgreSQL？

- 数据规模小：几百物品、几千留言、几万状态记录
- 零运维：不需要单独跑数据库服务
- 备份简单：`cp database.db backup.db`
- 群晖 NAS 资源富余，但简单胜过过度工程

### 4.4 为什么用 Cloudflare Tunnel 而不是端口映射？

- ✅ 无需公网 IP（很多学校宽带没有）
- ✅ 无需备案（Cloudflare 在境外）
- ✅ 自动 HTTPS + DDoS 防护
- ✅ 永久免费（无流量限制 for HTTP/HTTPS）

## 5. 性能与容量

| 指标 | 预估值 |
|------|--------|
| 同时在线用户 | < 100 |
| QPS（峰值） | < 10 |
| 数据库大小（1 年） | < 100 MB |
| 后端内存占用 | ~150 MB |
| SSE 连接数 | ~50（够用） |

群晖 SA6400 配置（AMD EPYC 12 核 / 大内存）**严重过剩**，可同时跑 N 倍负载。
